\section{Gambaran Umum}
Artifact ini menyetel \emph{seberapa banyak pod boleh dimatikan sekaligus} saat gangguan sukarela
(rolling update, drain node, scale-down) di Kubernetes, agar tidak pernah memecah \emph{kuorum}
cluster BEAM (Elixir terdistribusi). Intinya satu ide: \textbf{PodDisruptionBudget (PDB) yang
biasanya statis dan disetel manual, di sini diturunkan dari kuorum runtime}.

Komponennya kecil dan terpisah tugas:
\begin{itemize}[nosep]
  \item \code{QuorumBudget.Quorum} --- \emph{policy murni}: menghitung budget (\code{minAvailable},
        \code{maxUnavailable}) dari ukuran cluster $N$ dan ambang kuorum $Q$. Tanpa efek samping,
        mudah diuji.
  \item \code{QuorumBudget.Cluster} --- membaca keanggotaan cluster (\code{Node.list/0}) dan
        menurunkan $Q$ (mayoritas atau floor).
  \item \code{QuorumBudget.QuorumProbe} --- \emph{sensor read-only} (\code{GenServer}) yang melaporkan
        $N$, $Q$, dan apakah node sedang berkuorum.
  \item \code{QuorumBudget.PDBOperator} --- \emph{operator} Kubernetes: membaca probe tiap pod lalu
        menambal \code{minAvailable} PDB lewat \code{kubectl}.
  \item \code{QuorumBudget.QuorumWorkload} --- beban kerja yang bergantung-kuorum (commit hanya bila
        melihat kuorum), untuk studi kasus RQ8: mengukur konsekuensi nyata saat kuorum pecah.
  \item \code{QuorumBudget.Disruptor} --- mesin eksperimen: mensimulasikan rolling update sungguhan
        pada cluster peer dan \emph{mengukur} ukuran cluster tersisa + survivor yang terblokir.
  \item \code{QuorumBudget.Harness} --- membungkus pengukuran tiap RQ menjadi baris CSV.
\end{itemize}

\paragraph{Kunci desain: patok kuorum ke ukuran yang DIINGINKAN.} \code{Cluster.quorum\_threshold/1}
menghitung mayoritas dari \emph{ukuran cluster yang diinginkan} (jumlah replika Deployment, dari config
\code{:cluster\_size}), BUKAN ukuran live. Kalau memakai ukuran live, sebuah partisi yang menyusut akan
selalu mengira \code{majority(ukurannya sendiri)} terpenuhi --- yakni bug split-brain yang justru ingin
dicegah. Inilah satu baris yang membuat pengendali aman dan stabil saat membership berkedip (RQ10).

\section{Policy Kuorum (\code{quorum.ex})}
Ini jantung yang dapat dibuktikan. Rumusnya:
\begin{center}
\code{minAvailable = max(Q, q\_min)} \quad dan \quad \code{maxUnavailable = min(N - minAvailable, cap)}
\end{center}
\code{minAvailable = Q} menegakkan invarian keselamatan: anggota tersedia $A$ tak pernah turun di
bawah kuorum. \code{maxUnavailable} adalah ``ruang di atas kuorum'', dibatasi lagi oleh kapasitas
handoff \code{cap} (kopling ke pengendali grace M3: jangan matikan lebih banyak pod daripada yang
state-nya bisa dipindah tepat waktu).

\begin{lstlisting}
def budget(reading, opts \\ []) do
  n = reading.n; q = reading.q
  q_min = Keyword.get(opts, :q_min, 1)
  cap = Map.get(reading, :cap, n)
  min_available = q |> max(q_min) |> min(n)
  max_unavailable = (n - min_available) |> min(trunc(cap)) |> max(0)
  %{min_available: min_available, max_unavailable: max_unavailable}
end
\end{lstlisting}

Fungsi pendukung: \code{majority(n) = div(n,2)+1} (kuorum mayoritas), \code{violates\_quorum?(a,d,q)}
(apakah mengusir $d$ dari $a$ melanggar $Q$), dan \code{admit\_eviction(available, min\_available)}
(keputusan izinkan/tolak satu eviction --- inilah yang ditegakkan PDB Kubernetes).

\section{Membaca Cluster (\code{cluster.ex})}
\code{size/0} mengembalikan \code{length(Node.list()) + 1} (peer yang terlihat + diri sendiri).
\code{quorum\_threshold/1} menghitung $Q$: mayoritas dari ukuran hidup, atau bilangan tetap bila
beban kerja (mis. \code{:global}/CRDT) mendeklarasikan floor lewat konfigurasi.

\section{Probe \& Operator (\code{quorum\_probe.ex}, \code{pdb\_operator.ex})}
Probe adalah \code{GenServer} yang, saat ditanya, melaporkan map \code{\%\{n, q, cap, in\_quorum\}}.
Endpoint HTTP \code{GET /probe} (di \code{probe\_http.ex}) hanya membungkusnya sebagai JSON.

Operator berjalan di pod terpisah (peran \code{operator}, dipilih env \code{QUORUM\_ROLE}). Tiap 5
detik ia: (1) mendaftar IP pod app via \code{kubectl}, (2) membaca \code{/probe} tiap pod, (3)
menghitung \code{minAvailable} (mengambil yang TERBESAR antar-pod --- paling aman bila pod sedang tak
sepakat soal ukuran cluster), lalu (4) menambal \code{spec.minAvailable} PDB. Semua dibungkus
\code{try/catch}: error API sementara hanya di-log, operator tak crash (fail-safe).

\section{Mesin Eksperimen (\code{disruptor.ex})}
Untuk membuktikan klaim keselamatan dengan \emph{pengukuran nyata} (bukan aritmetika), \code{Disruptor}
membentuk cluster $n$ peer dan menjalankan rolling update: tiap batch (= ukuran budget) benar-benar
diputus dari cluster (\code{Node.disconnect}), lalu ukuran cluster tersisa DIUKUR dari
\code{Node.list/0} sebuah node survivor, lalu disambung kembali. Metrik: \code{min\_available} (ukuran
terkecil terukur), \code{quorum\_violated?} ($< Q$), dan \code{maintenance\_ms}.

\section{Menjalankan Eksperimen per RQ}
\begin{description}[leftmargin=1.5em]
  \item[RQ1 (safety) \& RQ2 (efficiency)] cluster peer nyata:
    \begin{lstlisting}[numbers=none]
MIX_ENV=test elixir --name primary@127.0.0.1 --cookie ck -S mix run harness/run.exs
    \end{lstlisting}
    Menulis \code{data/results\_safety.csv} dan \code{results\_efficiency.csv}. Hasil: budget statis
    memecah kuorum di tiap ukuran; quorum-aware tak pernah; dan quorum-aware $\sim$2,9$\times$ lebih
    cepat daripada konservatif (satu-per-satu).
  \item[RQ3--RQ6, RQ10 (policy murni)] \code{MIX\_ENV=test mix run harness/policy.exs} ---
    \code{results\_scale.csv} (\code{minAvailable} melacak mayoritas; hitung $O(1)$ $\sim$50\,ns),
    \code{results\_sensitivity.csv} (under-estimasi $Q$ tak aman $\to$ bias ke atas),
    \code{results\_flap.csv} (RQ10: patok ukuran live $\to$ 28 patch + 14 sampel tak-aman; patok ukuran
    diinginkan $\to$ 0/0).
  \item[RQ7 (kind)] \code{bash code/k8s/rq7\_eviction.sh} --- pada cluster kind hidup, PDB sadar-kuorum
    (\code{minAvailable}=$Q$) membuat eviction API MENOLAK eviction yang memecah kuorum; PDB
    tak-sadar-kuorum mengizinkannya.
  \item[RQ8 (workload)] \code{... -S mix run harness/workload.exs} --- beban kerja bergantung-kuorum di
    tiap node; budget statis memblokir $\lfloor N/2\rfloor$ survivor (tak bisa commit walau "hidup"),
    quorum-aware 0. \code{results\_workload.csv}.
  \item[RQ9 (baseline reactive)] \code{... -S mix run harness/reactive.exs} --- baseline yang menaikkan
    \code{minAvailable} setelah tiap pelanggaran memecah kuorum 4$\times$ sebelum konvergen; model-based
    0. \code{results\_reactive.csv}.
\end{description}

\section{Penerapan Kubernetes}
Manifes di \code{code/k8s/}: \code{rbac.yaml} (hak-paling-kecil: operator boleh baca pod +
baca/tambal PDB), \code{app.yaml} (Deployment $N$ replika + headless Service + PDB awal), dan
\code{operator.yaml} (Deployment operator). Satu image (\code{Dockerfile}) dipakai dua peran,
dibedakan env \code{QUORUM\_ROLE}.
